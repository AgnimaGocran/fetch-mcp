# TypeScript checklist

## 📏 Размер и структура файлов
1. В стандартном файле не более 300 строк (включая импорты).  
2. В файле со встроенными тестами не более 500 строк.  
3. В отдельном тестовом файле не более 600 строк.  
4. Файл, содержащий публичную структуру или перечисление, называется строго по имени типа: `PascalCase` → `snake_case.ts` (например, `UserProfile` → `ts`).
5. **Один файл — один основной публичный тип.** Вспомогательные типы допускаются только если они принадлежат исключительно этому типу и занимают `<30 строк`.  
6. Для подмодулей используется иерархическая структура директорий (`app/helpers/index.ts` + `app/helpers/*.ts` или просто `app/helpers.ts`), а не плоский список файлов с префиксами/суффиксами (`app_helpers.ts`), но при этом нельзя размещать файлы в подмодулях, если там ровно один файл! Например, ровно один файл `project.ts` или `index.ts` нельзя размещать в папке как `project/project.ts` или `project/index.ts`, т.е. промежуточных "одинарных" папок не должно быть!

## 🏗️ Архитектура и модульность
7. Крупный функционал вынесен в отдельные подпакеты.  
8. Слои представления (UI), бизнес-логики и работы с данными разделены и взаимодействуют через чёткие контракты.  
9. В файлах с высокоуровневой логикой отсутствуют низкоуровневые детали реализации и прямые системные вызовы.  
10. Модули и типы следуют принципу единственной ответственности: каждый модуль отвечает за свою предметную область/слой, каждая сущность — за одну задачу.  
11. Дублирование кода минимизировано: логика, повторяющаяся **3 и более раз**, выносится в общий модуль. Ранняя абстракция не допускается.  
12. Утилиты и хелперы вынесены в отдельные файлы, привязанные к конкретному домену или слою. Глобальные «мусорные» `utils.ts` / `helpers.ts` запрещены.
13. **В большинстве случаев используются готовые библиотеки вместо самописных алгоритмов и реализаций.**

## ⚡ Потоки, события и состояние
14. **Централизация состояния и событий в рамках группы функционала.** В пределах каждой логической группы (фичи, модуля, компонента, сервиса) состояние и обработка связанных событий имеют строго один источник. Мутации данных и реакция на события происходят только в одном месте группы. Глобальные монолитные хранилища запрещены, рассинхронизация данных не допускается.  
15. **Длительные и блокирующие операции не выполняются в основном потоке (GUI/событий)** (используется `async/await`, каналы или воркеры).

## ⌚ Прочее
16. В файлах для отступов используется только табуляция.
17. В операторах `if`, `else`, `for`, `while` во всех случаях стоят фигурные скобки (сокращённый синтаксис запрещён!).
18. В каждом проекте (включая подпроекты) в корне размещён следующий `.editorconfig` файл:
```
[*.ts,*.tsx]
indent_style = tab
indent_size = tab
tab_size = 4
insert_final_newline = false
trim_trailing_whitespace = true
```
19. В проектах используется Bun.js, а не Node.js.
20. В проектах не используется JavaScript (он строго настрого запрещён), а так же нетипизированный TypeScript.
21. Типы **any** и **unknown** используются с предельной осторожностью в иключительных случаях, каждое такое использование должно быть прокомментировано.
22. Для tsx файлов используется возврат jsx вёрстки без круглых скобок!
23. При использовании фреймворка Next.js (если данный пункт применим к проекту) используется App Router.
24. git репозиторий присутствует и инициализирован (если его нету, то инициализировать, в противном случае ничего не делать).
25. Mise En Place файл `mise.toml` есть (добавляем только если ранее он не был добавлен) в корне git репозитория со скриптами запуска (если ранее уже был mise.toml, то НЕ ДОБАВЛЯЕМ, НЕ ИЗМЕНЯЕМ скрипты!) `mise dev` => `bun run dev`,  `mise build` => `bun build`, `mise start` => `bun start`.
26. В проекте не используются устаревшие версии (особенно включая LTS!, используются строго последние версии!, старые LTS версии запрещены!), т.е. используются все самые последние свежие версии.
27. Отсутствуют ошибки и предупреждения линтера, компилятора.
28. Кавычки в TypeScript/JavaScript строках используются классические одинарные! При этом в HTML используются классические двойные для атрибутов. Внутри фигурных скобок для атрибутов TSX/JSX мы используем TypeScript/JavaScript и следовательно одинарные кавычки!
29. При наличии файлов тестов (хотя бы одного) к конкретному файлу для его тестирования, то необходимо тот час же все эти файлы разместить в папке-модуле (или вынести в отдельный проект с тестами, в зависимости от специфики проекта), например, у нас есть файл `fetcher.ts` и набор тестов `fetcher.simple.test.ts` и `fetcher.extended.test.ts`, то необходимо все эти файлы разместить в папке-модуле `fetcher`! А основной файл `fetcher.ts` сделать файлом `index.ts` и положить в корень этой папки `fetcher`.
30. Для обращения к модулям проекта - не использовать относительные пути! Кроме случаев обращения к модулям/подмодулям ниже по иерархии. Использовать `compilerOptions.paths` в `tsconfig.json` с префиксами `@` или `~`!
31. В проекте не используются устаревшие (depreciated) методы!
